# Ghost Theme Packaging And Validation

Use this skill for release, packaging, GitHub Actions, ZIP artifact, and GScan changes.

## Official Resources

- Ghost theme overview and GScan CLI docs: https://docs.ghost.org/themes and https://docs.ghost.org/themes/gscan/
- GScan web validator: https://gscan.ghost.org/
- GScan source and library usage: https://github.com/TryGhost/gscan
- Theme structure and required files: https://docs.ghost.org/themes/structure

## Package Flow

- The uploadable Ghost theme ZIP is generated by `grunt compress`.
- The expected ZIP path is `dist/atilla-rw-custom.zip`, derived from `package.json` `name`.
- `dist/` is ignored by git. Do not commit generated ZIP files.
- CI should build the ZIP on pull requests and pushes so GScan validates the exact uploadable artifact.
- CI should upload the ZIP artifact only after merged changes land on `master`.

## Local Commands

1. Install dependencies with `npm ci` if needed.
2. Build compiled assets with `npm run build`.
3. Package the theme with `npm run zip`.
4. Validate the ZIP with `npm run gscan`.

## CI Expectations

- Pull requests should run build, verify generated assets are committed, package the ZIP, and run GScan.
- Pushes to `master` should do the same, then upload `dist/atilla-rw-custom.zip` with `actions/upload-artifact`.
- Avoid committing ZIPs back to `master`; binary artifacts create noisy history and can trigger extra CI runs.
- Keep GitHub Action permissions minimal. Artifact upload only needs read access to repository contents.

## ZIP Contents

- Include runtime theme files: `assets/`, `partials/`, templates, locales, `package.json`, `LICENSE`, and `README.md`.
- Exclude repo and build-only files: `src/`, `dist/`, `.git/`, `.github/`, `.agents/`, `.codex/`, `node_modules/`, `AGENTS.md`, `Gruntfile.js`, and `package-lock.json`.
